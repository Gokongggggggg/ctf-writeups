>[!note]-  Understanding `printf` (Stack Behavior)
>
>Let’s start from a normal case:
>
>```c
>printf("%p", &x);
>```
>
>We can model this as:
>- **arg1** → pointer to `"%p"`
>- **arg2** → `&x`
>
>---
>
>### How Arguments Are Placed (Stack)
>
>Before `printf` is called :
>
>```
>push arg2   → &x
>push arg1   → "%p"
>call printf
>```
>
>📌 Stack (grow downward):
>```
>[arg2]   ← higher address
>[arg1]   ← lower address (top of stack at entry)
>```
>
>---
>
>### Special Role of `arg1`
>
>Inside `printf`:
>
>- **arg1 is always treated as a pointer**
>- It will be **dereferenced**
>
>So:
>```c
>arg1 → address of "%p"
>```
>
>becomes:
>```
>read memory at arg1 → '%'
>```
>
>👉 This is why:
>- `"%p"` is not the string itself
>- but a **pointer to memory containing `%p`**
>
>---
>
>### Internal Argument Pointer
>
>`printf` does NOT pop values from the stack.
>
>Instead:
>- it keeps an internal pointer (like `va_list`)
>- and moves forward as needed
>
>📌 On 32-bit:
>- each step = **+4 bytes**
>
>So conceptually:
>```
>arg1 → format string
>arg2 → next value
>arg3 → next value
>...
>```
>
>---
>
>### What Happens When `%p` Is Found
>
>From the format string:
>```
>"%p"
>```
>
>After dereferencing arg1:
>
>1. `printf` reads `'%'`
>2. sees `'p'`
>3. understands:  
>   👉 "read next value from arguments"
>
>So it:
>- moves internal pointer (that pointer for read from stack ) **+4 bytes**
>- reads that value (arg2)
>- prints it
>
>📌 Important:
>- `printf` **only reads**, it does NOT pop
>- it just keeps moving forward
>
>Also:
>- `%p` does NOT guarantee a "real address"
>- it prints **whatever 4 bytes it reads** from stack value 
>
>---
>
>### Now In Our Case
>
>```c
>printf(__format);
>```
>
>- Only **arg1 exists**
>- No arg2, arg3, etc.
>
>But:
>- arg1 is still treated as a pointer
>- so it gets dereferenced
>
>And:
>```c
>// from car_menu() , we see : ( _format will contain addr to heap (return from malloc , if we dereference it will contain out input ))
>__format = malloc(...)
>read(0, __format, ...)
>```
>
>👉 Meaning:
>- we fully control the contents of `__format`
>
>---
>
>### Exploit Behavior
>
>If we input:
>```
>%p %p %p %p %p ...
>```
>
>Flow:
>
>1. `printf` dereferences `arg1` → gets our string
>2. sees `%p`
>3. tries to read next argument
>
>But since no arg2:
>👉 it reads **whatever is next on the stack**
>
>Then:
>- moves +4 bytes
>- reads again
>- repeats
>
>→ effectively **walking the stack**
>
>---
>
>📌 Result:
>- We can leak stack values
>- eventually reaching `local_3c` (flag)
>

>### But `printf` Creates a New Stack Frame
>
>When we do:
>
>```c
>printf(__format);
>```
>
>execution leaves `car_menu()` and enters `printf()`.
>
>That means we now have **two different frames**:
>
>```text
>higher address
>────────────────────────
>car_menu() frame
>  [ebp-0x38]  local_3c
>  [ebp-0x40]  __format
>  saved EBP
>  return address to caller of car_menu
>────────────────────────
>printf() frame
>  saved EBP
>  local variables / internals
>  ap / va_list-style anchor
>────────────────────────
>lower address
>```
>
>---

>[!note]-  Where Does `printf` Start Reading? 
>
>The important question is:
>
>👉 when we use `%p`, **where does `printf` actually read from?**
>
>---
>
>### What Our Input Is Used For
>
>For example:
>```
>%p %p %p %p
>```
>
>Our input is **not treated as data to print**.
>
>👉 Instead, it is used as **instructions**:
>
>- `%p` → "read one value"
>- `%p` → "read next value"
>- and so on
>
>So conceptually:
>
>```
>[our input already used as instructions]   ← parsed (%p, %p, ...)
>[ ??? ]                                   ← starting point
>[ next ]
>[ next ]
>...
>```
>
>👉 This `[ ??? ]` is where `printf` begins reading
>
>---
>
>### So What Happens?
>
>Even though we only provided one input,
>`printf` still assumes:
>
>```
>"there should be more values here"
>```
>
>So it starts reading from that starting point.
>
>---
>
>### How It Moves
>
>Each `%p`:
>- reads one value
>- then moves forward
>
>On 32-bit:
>```
>+4 bytes each step
>```
>
>So:
>```
>read → move → read → move → ...
>```
>
>---
>
>### Why This Reaches the Flag
>
>This reading continues upward through memory:
>
>- data inside `printf`
>- return-related data
>- then back into `car_menu`
>
>Eventually reaching:
>
>```
>ebp - 0x38 → local_3c (flag)
>```
>
>---
>

>[!note]-  How Many `%p` Do We Need?
>
>In practice, the easiest way is just to **spam `%p` gradually**.
>
>For example, start with:
>```
>%p %p %p %p %p %p %p %p %p %p
>```
>
>If the flag is not there yet, just add more:
>```
>%p %p %p %p %p %p %p %p %p %p %p %p %p ...
>```
>
>So the workflow is simply:
>- try a few `%p`
>- check the output
>- if the flag is not reached yet, add more
>
>📌 Since this is a simple challenge, brute-forcing the offset like this is much easier than trying to calculate it exactly.

>
>After increasing the number of `%p`, the leak finally reaches the beginning of the flag:

>
>```text

>0x5f643164 → d1d_

>
>```text
