## Source code Analysis

> [!note]- Explanation
> From the source code, we can immediately see that this challenge is very similar to the previous one.
> 
>![[Pasted image 20260318192435.png]]
>
> ```c
> char buf[32];
> ```
>
> However, unlike the previous challenge that used `read()`, this one uses:
>
> ```c
> fgets(buf, 128, stdin);
> ```
>
> At first glance, `fgets()` looks safer because it allows the programmer to specify a maximum number of bytes to read. But the issue here is that the specified size is **128 bytes**, while the actual buffer can only hold **32 bytes**.
>
> This means the program still accepts input far beyond the buffer’s capacity, resulting in a **stack buffer overflow**. So even though the input function changed from `read()` to `fgets()`, the bug remains essentially the same: the input limit is larger than the destination buffer.
>

## Disas Vuln()


> [!note]- Explanation
> To determine the buffer offset, we can inspect the disassembly of `vuln` in `gdb` using `disas vuln`.
>
>![[Pasted image 20260318193337.png]]
>
> The important instruction here is:
>
> ```asm
> lea eax, [ebp-0x28]
> ```
>
> This shows that the input buffer starts at **`ebp-0x28`**, meaning the buffer offset is **`0x28` bytes** from the base pointer.
>
> A bit more detail:
>
> - `sub esp, 0x24` allocates local stack space
> - there is also saved padding/alignment such as the preserved `ebx`
> - as a result, the compiler places the buffer at `ebp-0x28`
>
> So even though the declared buffer is:
>
> ```c
> char buf[32];
> ```
>
> the actual stack offset to reach the saved frame area is **40 bytes (`0x28`)**, not just 32 bytes.
>
> This is why we use **`0x28` bytes of padding** before overwriting the saved return address.

## Solve.py

>[!note]- Explanation
>The payload uses **44 bytes** before the target address because we need to fully reach the saved return address on the stack.
>
>![[Pasted image 20260318193828.png]]
>
>From the disassembly of `vuln`, we already identified that the buffer starts at:
>
>```asm
>lea eax, [ebp-0x28]
>```
>
>This means the distance from the start of the buffer to the base pointer is **`0x28` bytes = 40 bytes**.
>
>After those 40 bytes, the next value on the stack is the **saved EBP**, which takes **4 bytes** on a **32-bit** binary.
>
>So the total offset becomes:
>
>- **40 bytes** to fill the buffer up to saved EBP
>- **+ 4 bytes** to overwrite saved EBP
>- then the next 4 bytes land on the **return address**
>
>That is why the payload uses:
>
>```python
>b"A" * 44
>```
>
>In other words, **44 bytes** are needed so the next value we write will overwrite `EIP` with the address of `win()`.
>
>On a **64-bit** binary, this is often **+8** instead of **+4** because the saved base pointer (`RBP`) is 8 bytes wide. But since this challenge is **32-bit**, the saved `EBP` is only **4 bytes**.
