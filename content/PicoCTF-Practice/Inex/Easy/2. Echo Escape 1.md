## Vulnerability

> [!note]- Explanation
> This results in a **buffer overflow vulnerability**.
>
> The function `read()` does not perform bounds checking and will write up to 128 bytes into a 32-byte buffer.
>
> As a result, this allows overwriting adjacent memory such as:
> - Saved base pointer (RBP)
> - Return address (RIP)
>
> This gives us control over the program execution flow.

## Reading the Disassembly of `main()`

> [!note]- Explanation
> Before calling `read()`, the program prepares its arguments in registers.
>
> On x86-64 Linux, function arguments are passed using registers in this order:
> - `rdi` → 1st argument
> - `rsi` → 2nd argument
> - `rdx` → 3rd argument
> - `rcx` → 4th argument
>
> In the disassembly, we see:
>
> ![[Pasted image 20260318175814.png]]
>
> This corresponds to:
>
> ```c
> read(0, buf, 128);
> ```
>
> The setup works as follows:
>
> - `lea rax, [rbp-0x20]`  
>   Loads the address of the local buffer into `rax`
>
> - `mov edx, 0x80`  
>   Sets the 3rd argument to `0x80` (128), which is the number of bytes to read
>
> - `mov rsi, rax`  
>   Sets the 2nd argument to the buffer address
>
> - `mov edi, 0x0`  
>   Sets the 1st argument to `0`, which means input is read from `stdin`
>
> - `call read@plt`  
>   Calls `read(0, buf, 128)`
>
> From `lea rax, [rbp-0x20]`, we know the buffer starts at offset `rbp-0x20`, meaning it occupies 32 bytes on the stack.
>
> Since the saved `rbp` is located right above it, the offset to the return address becomes:
>
> ```
> 0x20 + 0x8 = 0x28
> ```
>
> So, **40 bytes** are needed to overwrite.

>[!note]- Explanation
> Explanation of the payload:  
> - `b"A" * 32` fills the buffer  
> - `b"B" * 8` overwrites the saved RBP  
> - `p64(win_addr)` overwrites the saved return address with the address of `win()`  
>  
> This works because the buffer is 32 bytes long, and the saved RBP is located directly after it on the stack.  
> Therefore, after 40 bytes, the next value written will overwrite the return address.  
>  
> Once `main()` finishes and returns, execution jumps to `win()` instead of the normal return location, which causes the program to print the flag.
