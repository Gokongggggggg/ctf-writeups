## Hash()

``` python
leaked_bytes = [105, 85, 98, 104, 56, 49, 33, 106, 42, 104, 110, 33]
h = 5381
for b in leaked_bytes:
    h = ((h * 33) + b) & 0xFFFFFFFFFFFFFFFF
print(f"Hasil Hash yang Benar: {h}")
```

>[!note]- Explanation
>   ```python
>   leaked_bytes = [105, 85, 98, 104, 56, 49, 33, 106, 42, 104, 110, 33]
>   h = 5381
>   for b in leaked_bytes:
>       h = ((h * 33) + b) & 0xFFFFFFFFFFFFFFFF
>   print(h)
>   ```
>
> - Why do we need:
>   ```python
>   & 0xFFFFFFFFFFFFFFFF
>   ```
> - Because in C, this hash runs inside a **64-bit integer**
> - A 64-bit value can only keep the lowest **64 bits**
> - If the value grows larger, the extra upper bits are **discarded automatically**
>
> - `0xFFFFFFFFFFFFFFFF` means:
>   - 16 hex `F`s
>   - each hex `F` = 4 bits of `1`
>   - `16 × 4 = 64` bits
> - So this mask is basically:
>   - “keep only the lowest 64 bits”
>
> - Why is this important?
> - If we naively run the formula in Python:
>   ```python
>   h = h * 33 + b
>   ```
>   Python integers keep growing with **unlimited precision**
> - But C does **not** do that here
> - In C, once the value overflows past 64 bits, it wraps around immediately
>
> - So the difference is:
>   - **C** → overflow happens at every step
>   - **Python** → no overflow unless we manually simulate it
>
> - That is exactly why we apply the mask on every iteration:
>   ```python
>   h = ((h * 33) + b) & 0xFFFFFFFFFFFFFFFF
>   ```
> - This mimics C’s 64-bit wraparound behavior
>
> - Why is it safe even before overflow?
> - Because if `h` is still smaller than 64 bits, masking changes nothing
> - It only matters once the value exceeds the 64-bit limit
>
> ### Conclusion
> - We must mask with `0xFFFFFFFFFFFFFFFF` on **every round**
> - Otherwise Python’s result will not match the real C hash
