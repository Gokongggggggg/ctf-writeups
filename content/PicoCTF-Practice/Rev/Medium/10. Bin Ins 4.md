>[!note]- Explanation
> - In Ghidra, we see references to:

> __static_initialization_and_destruction_0  
> ```  
>  
> - This indicates that `flagParts` is **initialized before `main()` runs**  
>  
> - More precisely:  
> - it is **set up during static initialization phase**  
> - meaning the program fills `flagParts` *before entering `main`*, not dynamically later  
>  
> - So instead of relying on runtime behavior (like socket / send), we can:  
> - jump into `__static_initialization_and_destruction_0`  
> - trace how `flagParts` is assigned
## Understanding how `flagParts` is filled

![[Pasted image 20260320052624.png]]
![[Pasted image 20260320052943.png]]
>[!note]- Explanation  
> - At the top we see:  
> ```asm  
> LEA RSI, [flagParts]  
> ```  
> - This means **RSI holds the address of `flagParts`**  
>  
> - Then we see repeated patterns like:  
> ```asm  
> LEA param_2, [s_...]  
> MOV param_1, RSI  
> CALL std::__cxx11::string::string  
> ```  
>  
> - Important mapping (x64 calling convention):  
> - `param_1` → **RCX** (destination object)  
> - `param_2` → **RDX** (input data)  
>  
> - So what actually happens:  
> - `param_1 (RCX)` = address of `flagParts`  
> - `param_2 (RDX)` = pointer to a constant string (`s_...`)  
>  
> - Meaning each call is basically:  
> ```cpp  
> flagParts[i] = "some_chunk";  
> ```  
>  
> - To get the flag:  
> - collect all `s_...` values in order  
> - concatenate them into one full string  
> - decode the result using **base64**
